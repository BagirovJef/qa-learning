## 1. ИНСТРУМЕНТЫ

### Swagger
**Определение:** Swagger — инструмент для проектирования, документирования и тестирования RESTful API. Позволяет описывать API в формате OpenAPI.

**Применение:**
- Использую как документацию — смотрю эндпоинты, параметры, схемы ответов
- Проверяю, соответствует ли API документации
- Если есть UI (Swagger UI) — отправляю тестовые запросы прямо оттуда
- Создаю тестовые данные, когда DevOps не может раздать окружение

**Пример:** Swagger UI позволяет проверить, соответствует ли API документации, и быстро отловить несоответствия. Делают уникального клиента и проводят все тестовые сценарии.

### Postman
**Определение:** Postman — инструмент для тестирования API, позволяющий отправлять HTTP-запросы, проверять ответы и автоматизировать тесты.

**Применение:**
- Создание и отправка запросов (GET, POST, PUT, DELETE и др.)
- Автоматизация тестов через коллекции и скрипты (JavaScript)
- Валидация ответов (статус-коды, JSON-схемы, время ответа)
- Генерация документации и мокирование API
- Тестирование на dev-окружении сразу после выкатки фичи
- Быстрое позитивное и негативное тестирование

**Типы данных в Postman:**
- int: целочисленный тип (пример: 30)
- float: тип с плавающей точкой (пример: 1.75)
- string: текстовый тип (пример: "John Doe")
- bool: логический тип true/false (пример: депозит True/False)

**Пример:** Коллекции в Postman позволяют быстро протестировать все эндпоинты после выкатки фичи на dev-окружение. Если находим баг, разработчик чинит его в течение часа или дня.

### Kibana
**Определение:** Kibana — инструмент для визуализации и анализа логов, особенно в связке с Elasticsearch.

**Применение:**
- Мониторинг ошибок и поиск паттернов в логах
- Создание дашбордов для отслеживания метрик
- Поиск ошибок по статус-кодам (например, 500)
- Настройка алертов через Watcher
- Просмотр всех запросов на этапе регресса

**Пример:** Запрос в Kibana для поиска ошибок 500: { "query": { "match": { "status": "500" } } }. Использую Trace ID — вставил и быстро посмотрел, в каких сервисах используется.

### Kafka
**Определение:** Kafka — распределённая система обмена сообщениями для построения потоковых данных.

**Применение:**

- Контроль дублирования и потери сообщений
- Тестирование устойчивости к падениям сервисов
- Нагрузочное тестирование топиков
- Проверка в UI — ушло сообщение или нет

**Пример:** Использую Kafkacat и UI-клиенты (Kowl, Conduktor) для мониторинга топиков. Проверяю, что события уходят в нужные топики, смотрю, не дублируются ли сообщения.

### Fiddler
**Определение:** Fiddler — программа для перехвата и изменения трафика между клиентом и сервером. Выступает посредником (прокси).

**Применение:**
- Перехват запросов с мобильных устройств
- Подмена ответов сервера для тестирования граничных случаев
- Проверка заголовков и тела запросов
- Анализ утечек данных
- Проверка наличия HTTPS везде, где требуется
- Отсутствие передач паролей в открытом виде

**Пример:** Использую для мобильного тестирования. Для HTTPS использовал декрипт трафика через предустановленный сертификат Fiddler.

### Charles
**Определение:** Charles — прокси-сервер для перехвата и анализа HTTP/HTTPS трафика.

**Применение:**
- Перехват и анализ запросов/ответов (REST API, GraphQL)
- Мокирование ответов (Map Local для тестирования edge-кейсов)
- Тестирование скорости загрузки (Waterfall, latency)
- Отладка мобильных приложений (прокси для iOS/Android)
- Throttling — эмуляция медленного соединения (3G, Edge)
- Breakpoints — изменение запросов на лету

**Пример:** Map Local: перехватываю запрос к /api/users → Tools → Map Local → Add → указываю локальный JSON-файл. Breakpoints: подмена status_code на 500 для проверки обработки ошибок.

### Kubernetes
**Определение:** Kubernetes (K8s) — система автоматизации развёртывания и управления контейнеризированными приложениями. Управляет десятками/сотнями контейнеров.

**Применение:**

- Проверка конфигураций и переменных окружения
- Перезапуск упавших контейнеров

**Пример:** Ситуация: POST /api/orders/create возвращает 500. Через Lens захожу в namespace order-service, открываю логи. Вижу: "connection refused to payment-service". Иду в payment-service — ошибка: не загрузились переменные окружения. Передаю разработчикам точное описание.

### CI/CD
**Определение:** CI/CD — практика автоматизации сборки, тестирования и развёртывания кода. Разработчик пишет код → CI запускает тесты → если багов нет, CD выкладывает обновление.

**Применение:**
- Автотесты запускаются автоматически при каждом коммите
- Проверка отчётов о выполнении тестов в CI
- Анализ логов при падении тестов
- Настройка тестовых окружений для деплоя
- Быстрый фидбек по багам
- Снижение рисков при релизах

**Пример:** Jenkins, GitLab CI, GitHub Actions — инструменты CI/CD. Смотрю, что автотесты запускаются автоматически при каждом коммите, проверяю отчёты, если тесты упали — анализирую логи и прикрепляю баг-репорт.

## 2. ТЕСТИРОВАНИЕ

### Что такое тестирование
**Определение:** Тестирование — оценка текущего состояния системы, проверка соответствия ожидаемого поведения фактическому.

**Цели:**
- Выявление дефектов — обнаружение багов и несоответствий
- Обеспечение качества — проверка соответствия требованиям
- Повышение надёжности — минимизация рисков сбоев
- Валидация и верификация — подтверждение корректности работы
- Предоставление информации — отчёты о состоянии продукта

**Пример:** Тестирование формы входа: проверка корректного ввода логина/пароля, обработки ошибок.

---

### Виды тестирования
**Определение:** Виды тестирования классифицируются по целям и объекту проверки.

**Связанные с изменениями:**
- Smoke — быстрая проверка важного функционала ("сайт живой"). Если smoke прошёл успешно, проводим глубокое sanity тестирование.
- Sanity — более глубокая проверка конкретного функционала
- Regression — проверка, что новый функционал не сломал старый (авто и ручное)

**Нефункциональное:**
- Нагрузочное (Performance) — работа под нагрузкой
- Стресс — экстремальные условия
- Безопасности — поиск уязвимостей
- Usability — оценка UX/UI
- Локализации — адаптация под языки/регионы
- Совместимости — работа на разных ОС/браузерах
- Восстановления — проверка отказоустойчивости
- Масштабируемости
- Надёжности

**Функциональное:**
- Unit (модульное) — проверка одного компонента в изоляции
- Integration — проверка взаимодействия компонентов
- System — проверка собранной системы
- Acceptance — финальная проверка перед сдачей заказчику
- E2E — проверка работы приложения от начала до конца

**Пример:** A/B-тестирование — сравнение двух версий интерфейса. Тестирование установки — проверка корректности установки, обновления и удаления.

---

### Методы тест-дизайна
**Определение:** Методы тест-дизайна — техники для создания тестов с максимальным покрытием при минимальных затратах.

**Методы:**
- Эквивалентное разделение — делим данные на группы, проверяем по одному из каждой
- Граничные значения — проверяем на границах диапазонов (0, -1, +1)
- Таблица принятия решений — тестируем комбинации условий "Если... то..."
- Парное тестирование — проверка комбинаций параметров для сокращения тест-кейсов
- Диаграмма перехода состояний — тестирование смены состояний
- Предугадывание ошибок — тестирование на основе опыта и типичных багов (SQL-инъекции)
- Исследовательское тестирование — без сценариев, изучение на ходу

**Пример:** Поле "Возраст" от 18 до 65: проверяем 17, 18, 65, 66 (граничные значения).

---

### Тестовая документация
**Определение:** Тестовая документация — набор документов, описывающих процесс и результаты тестирования.

**Виды:**
- **Тест-план:** Описывает объект тестирования, график, критерии начала/окончания, стратегию, риски, ресурсы. Состоит из: оглавление, описание системы, функциональности, стек инструментов, виды тестирования, план работ, ссылки на баг-репорт и отчёт, критерии начала/окончания.
- **Чек-лист:** Список пунктов для проверки без детальных шагов. Google Docs для smoke-тестов перед релизом.
- **Тест-кейс:** Набор шагов для проверки функции (ID, название, описание, шаги, данные, ОР/ФР, приоритет, статус, автор). Набор тест-кейсов называется test suite.
- **Баг-репорт:** Заголовок, шаги воспроизведения, ФР, ОР, окружение, приоритет/серьёзность, скриншоты/логи/видео.
- **Отчёт по тестированию:** Итоги: какие тесты прошли/не прошли, сколько багов найдено и в каком статусе.
- **Методология тестирования:** Описание модели (Agile, Waterfall и т.д.).
- **Тестовое покрытие:** Степень, в которой требования или код охвачены тестами.

**Пример:** Баг-репорт: "Кнопка Submit не реагирует на клик" + шаги + скриншот + консоль. Документировали через скриншоты и скринкасты для сложных сценариев.

---

### Требования
**Определение:** Требования — условия, которым должна соответствовать система.

**Атрибуты качества:**
- Корректность — точно отражает потребности заказчика
- Однозначность — исключает двусмысленность ("быстро" → "≤2 сек")
- Проверяемость — можно объективно проверить тестами
- Полнота — охватывает все аспекты
- Непротиворечивость — не конфликтует с другими требованиями
- Атомарность — одно требование = одна функция
- Трассируемость — уникальный ID для отслеживания

**Виды:**
- Функциональные: что система делает (авторизация пользователя)
- Нефункциональные: как работает (производительность, безопасность, удобство)
- Бизнес-требования: цели бизнеса (увеличение продаж)
- Пользовательские: потребности пользователей (интерфейс, логика)

**Пример:** Функциональное: "При неверном пароле — ошибка". Нефункциональное: "1000 пользователей, отклик ≤2 сек". Проверял требование "отправка email при регистрации": что считается валидным email? Куда отправляется? Есть ли timeout?

---

### Когда начинать тестирование
**Определение:** Тестирование должно начинаться как можно раньше в SDLC, желательно на этапе анализа требований.

**Преимущества раннего тестирования:**
- Снижение стоимости исправления багов
- Лучшее понимание требований командой
- Возможность влиять на архитектуру продукта

**Пример:** По данным IBM, баг в 15-100 раз дороже исправить на этапе поддержки, чем на этапе проектирования.

---

### Уровни тестирования
**Unit (модульное):** Пишут разрабы (редко автоматизаторы), проверка одного компонента в изолирующих условиях.

**Integration:** Совмещаем несколько компонентов и проверяем взаимодействие.

**System:** Проверка полностью собранной системы на соответствие требованиям.

**Acceptance:** Финальная проверка перед сдачей продукта заказчику. Проверяем соответствие Acceptance plan.

**E2E:** Проверка работы приложения от начала до конца.

**Пример:** Unit: тестирование функции сложения. Integration: проверка API + БД. System: полный функционал. Acceptance: демонстрация заказчику.

---

### Smoke vs Regression
**Smoke:** Smoke — быстрая проверка основных функций после сборки. Цель — убедиться, что критичные компоненты работают перед глубоким тестированием.

**Regression:** Regression — повторное тестирование проверенных функций после изменений. Цель — убедиться, что новые правки не сломали существующий функционал. Часто автоматизируется.

**Разница:** Smoke — "сайт живой". Regression — "всё ещё работает".

---

### Стенды
**Dev Stand:** Тестируем саму разработку. Разработчики проверяют свои фичи. Может быть нестабильно, данные тестовые, можно всё ломать.

**PreProd (Stage):** Максимально идентичный Prod: столько же ресурсов, такие же базы, 1 в 1. Основная рабочая зона для функционального, интеграционного и регрессионного тестирования.

**Prod:** Реальная рабочая среда, где пользуются клиенты. Нагрузочное тестирование проводится в часы низкой нагрузки (после работы).

**Пример:** На Stage проводим регресс. На Dev — базовый позитив/негатив. На Prod — нагрузочное (но не с реальными деньгами, делается Dev-стенд для транзакций).

## 3. API

### REST vs SOAP
**Определение:** REST — архитектурный стиль на HTTP с JSON. SOAP — строгий протокол на XML.

**Сравнение:**
- REST: Гибкий, использует HTTP-методы (GET, POST), данные в JSON/XML, кэшируемый, масштабируемый.
- SOAP: Строгий протокол, только XML, встроенная безопасность, используется в банках и корпоративных системах.

**Пример:** REST: GET /api/users. SOAP: XML-конверт с envelope, header, body.

---

### HTTP-методы
**Определение:** HTTP-методы определяют тип операции над ресурсом.

**Методы:**
- GET — получить данные (безопасный, идемпотентный)
- POST — создать ресурс (не идемпотентный)
- PUT — заменить полностью (идемпотентный)
- PATCH — частично обновить (идемпотентный)
- DELETE — удалить (идемпотентный)

**Пример:** PUT /users/1 заменяет все поля. PATCH /users/1 обновляет только указанные поля.

---

### Безопасность методов
**Определение:** POST безопаснее GET для передачи конфиденциальных данных.

**Объяснение:** При GET данные передаются в URL (видны в истории, логах, закладках). При POST данные в теле запроса (не видны в URL).

**Пример:** Пароль через GET: /login?password=123 (видно!). Через POST: в теле запроса (скрыто).

---

### JSON vs XML
**Определение:** JSON и XML — текстовые форматы для передачи структурированных данных.

**JSON:** Лёгкий, читаемый, на основе JavaScript. Объекты {}, массивы [], ключи в кавычках. Нет комментариев.

**XML:** Язык разметки с тегами. Самодокументированный, поддерживает иерархию. Используется в SOAP.

**Пример:** JSON: {"name": "John", "age": 30}. XML: <user><name>John</name><age>30</age></user>

---

### Идемпотентность
**Определение:** Идемпотентность — свойство операции, при котором многократное выполнение даёт тот же результат, что и однократное.

**Методы:** Идемпотентные: GET, PUT, DELETE. Не идемпотентный: POST. PATCH — гибридный.

**Пример:** DELETE /users/1: первый раз удаляет, последующие — ничего не меняют (результат тот же).

---

### Виды тестирования API
**Функциональное:** Проверка что программа делает (бизнес-логика)

**Нагрузочное:** Проверка как себя ведёт (сколько пользователей выдержит, скорость)

**Безопасности:** Можно ли взломать, украсть данные, обойти авторизацию

**Интеграционное:** Как разные части системы работают вместе

**REST/SOAP:** Валидные запросы, структура сообщений, ошибочные сценарии

**Чек-лист:**
- Валидность и формат входных данных
- Ответ API соответствует ожидаемым значениям
- Правильные коды ответа (200, 401, 404)
- Аутентификация и авторизация работают корректно
- Бизнес-логика работает
- Безопасность (SQL-инъекции, XSS)
- Нет доступа к конфиденциальным данным

---

### PUT vs PATCH
**PUT:** PUT заменяет полностью данные. Если чего-то нет в body запроса — это поле удалится.

**PATCH:** PATCH лишь частично обновляет. Если чего-то нет в body запроса — это поле останется без изменений.

**Пример:** PUT /users/1: {"name": "John"} → все остальные поля удалятся. PATCH /users/1: {"name": "John"} → обновится только name.

---

## 4. SQL

### Основные операторы
**Определение:** SQL — язык структурированных запросов для работы с реляционными БД (MySQL, PostgreSQL).

**Операторы:**
- SELECT — выборка данных
- INSERT INTO — добавление записей
- UPDATE — обновление записей
- DELETE — удаление записей

**Пример:** INSERT INTO users (name, email) VALUES ("John", "john@example.com")

---

### JOIN
**Определение:** JOIN — оператор для объединения данных из двух и более таблиц.

**Виды:**
- LEFT JOIN: все записи из левой + совпадения из правой
- RIGHT JOIN: все записи из правой + совпадения из левой
- INNER JOIN: только совпадающие записи из обеих таблиц
- FULL JOIN: объединение ВСЕХ строк из обеих таблиц

**Пример:** SELECT * FROM orders LEFT JOIN users ON orders.user_id = users.id

---

### GROUP BY и HAVING
**Определение:** GROUP BY группирует результаты по столбцу. HAVING фильтрует группы после агрегации.

**Применение:** GROUP BY используется с агрегатными функциями (COUNT, SUM, AVG). HAVING заменяет WHERE для агрегатов.

**Пример:** SELECT department, COUNT(*) FROM employees GROUP BY department HAVING COUNT(*) > 5

---

### ORDER BY и DISTINCT
**Определение:** ORDER BY сортирует результаты. DISTINCT удаляет дубликаты.

**Применение:** ORDER BY применим к числам и строкам (ASC/DESC). DISTINCT возвращает уникальные значения.

**Пример:** SELECT DISTINCT city FROM users ORDER BY city ASC

## 5. АРХИТЕКТУРА

### Клиент-сервер
**Определение:** Клиент-сервер — модель, где клиент отправляет запросы, сервер обрабатывает и возвращает ответ. Бывает 2-х и 3-х звенной.

**Виды:**
- Тонкий клиент: минимум логики на клиенте, вся обработка на сервере (веб-приложения)
- Толстый клиент: большая часть работы на клиенте (десктопные/мобильные приложения)

**Запрос:** Запрос состоит из: метод, URL, параметры, тело. Тело передаётся в виде текста (REST: JSON/XML, SOAP: XML).

**Пример:** Браузер (клиент) → HTTP/HTTPS-запрос → Сервер → БД → Ответ → Браузер

---

### Микросервисы
**Определение:** Микросервисная архитектура — приложение как набор небольших независимых сервисов.

**Особенности:**
- Декомпозиция — сервисы разрабатываются и разворачиваются независимо
- Независимость — разные технологии для разных сервисов
- Коммуникация через API — REST, gRPC, асинхронно (Kafka, RabbitMQ)
- Автономность — масштабирование отдельных сервисов
- Устойчивость к сбоям — отказ одного не ломает всё

**Преимущества:**
- Гибкость в разработке — команды работают одновременно
- Масштабируемость — отдельные сервисы под нагрузкой
- Упрощённое развёртывание — деплой отдельно
- Лёгкость в поддержке — меньшие кодовые базы
- Технологическая независимость — разные технологии

**Пример:** Сервис заказов, сервис оплаты, сервис уведомлений — каждый развёртывается отдельно. Node.js + React, REST API + Kafka, PostgreSQL + Redis, Docker + Kubernetes.

---

### Синхронное vs Асинхронное
**Определение:** Синхронное — запрос-ответ сразу. Асинхронное — через очереди сообщений.

**Сравнение:**
- Синхронное: REST API: отправил запрос → получил ответ сразу. Проще, но сервисы связаны.
- Асинхронное: Kafka/RabbitMQ: отправил сообщение → обработается позже. Надёжнее при пиках.

**Пример:** Синхронно: GET /api/orders. Асинхронно: сообщение в топик "orders-created". У нас есть эндпоинты для отправки заказа — отправили в Парус, он сразу вернул данные. Асинхронных процессов (Kafka) нет.

---

### TCP/IP
**Определение:** TCP/IP — набор протоколов для передачи данных в сети.

**Протоколы:**
- TCP: разбивает данные на пакеты, гарантирует доставку, собирает обратно
- IP: отвечает за адресацию — знает, куда отправлять данные

**Пример:** TCP делит файл на кусочки, IP доставляет их по адресу, TCP собирает обратно.

---

### WebSocket
**Определение:** WebSocket — протокол для двусторонней связи в реальном времени.

**Применение:** Сервер может отправлять данные клиенту без запроса. Используется в чатах, играх, live-обновлениях.

**Пример:** Чат: сервер сам отправляет новое сообщение всем подключённым клиентам.

---

## 6. АВТОРИЗАЦИЯ

### Идентификация
**Определение:** Идентификация — процесс определения личности пользователя (ввод логина).

**Пример:** Ввод логина или email при входе на сайт.

---

### Аутентификация
**Определение:** Аутентификация — процесс проверки подлинности (ввод пароля, SMS, отпечаток).

**Пример:** Ввод пароля после логина. Двухфакторная: пароль + SMS.

---

### Авторизация
**Определение:** Авторизация — процесс определения прав доступа после успешной аутентификации.

**Пример:** После входа пользователь получает доступ к личному кабинету, но не к админке.

---

### Верификация vs Валидация
**Верификация:** Проверяет, что продукт соответствует ТЗ.

**Валидация:** Проверяет, что продукт соответствует ожиданиям пользователя.

**Пример:** Верификация: все ли функции из ТЗ реализованы. Валидация: удобно ли пользователю.

---

### Storage
**Определение:** Хранилища данных в браузере.

**Виды:**
- LocalStorage: данные доступны во всех вкладках, сохраняются после закрытия браузера. Нет срока действия.
- SessionStorage: данные только в одной вкладке, удаляются после закрытия вкладки.

**Пример:** LocalStorage: тема оформления. SessionStorage: данные текущей сессии.

---

### Cache vs Cookie
**Определение:** Механизмы хранения данных в браузере.

**Виды:**
- Cache: хранит файлы сайта (картинки, стили) для ускорения повторной загрузки
- Cookie: персонализированные данные (логин, настройки) для идентификации пользователя

**Пример:** Cache: логотип сайта. Cookie: "запомнить меня".

## 7. ПРОЦЕССЫ

### Баг-флоу
**Workflow:**
- To Do → задача создана
- In Progress → разработчик взял в работу
- Review → проверка кода коллегами
- Approve → код одобрен
- Testing → тестирование на DEV и Stage

**Процесс:**
- Тестирую по ТЗ (аналитика)
- Позитивные кейсы, негативные сценарии
- Проверки на null, кривые значения
- Проверка ошибок сервера
- Если баг — перепроверяю себя
- Иду к разработчику или PM уточнить
- Завожу баг-репорт с curl, ОР, ФР

**Пример:** Ожидаю 200 OK, получаю 400. Перепроверяю, иду к PM. Завожу баг-репорт: curl запрос, ожидаемый результат, полученный результат. Перекидываю на PM, он распределяет.

### Спринты и релизы
**Процесс:**
- Двухнедельные спринты (модифицированный Scrum, ScrumBan)

- Перед новым спринтом — релиз
- Стараются релизиться каждый спринт
- Важный функционал релизят ночью
- Адаптивная методология — что-то сдвигается

**Роли:**
- 3 разработчика: включая техлида (отвечает за тех. стек)
- 1 аналитик/тимлид: сеньор разраб, ставит аналитику, следит за процессом

- Архитектор: общий по всему Парусу

### Распределение задач
**Процесс:**
- Задачи берём самостоятельно из бэклога
- Крупные фичи распределяет Team Lead
- На дейликах рассказываю о прогрессе и рисках
- На планировании помогаю оценивать задачи
- Постоянно взаимодействую с разработчиками и аналитиками

**Когда:** Полный регресс — перед крупным релизом, за 1-2 дня до выкатки. Регресс после каждой фичи — всегда.

**Критерии:**

- Зона изменений — что меняли и что связано
- История багов — проблемные места

**Процесс:**
- На DEV — базовый позитив/негатив
- На Stage — пишем тест-кейсы для полного регресса
- Прогоняем все тест-кейсы
- Если находим ошибку — оформляем баг-репорт

**Состав:**
- 3 разработчика: включая техлида
- 1 аналитик/тимлид: сеньор разраб
- 1 PM: Product Manager
- 3 тестировщика: 2.5 на постоянной основе

- Архитектор: общий по нескольким проектам

**Процесс:**
- Разрабатываю тестовую документацию в TestIT
- Участвую в дейликах, планировании и ретро
- Взаимодействую с разработчиками и аналитиками

## 9. ДЕФЕКТЫ

### Ошибка/Баг/Дефект
**Ошибка (error, mistake):** действие человека, приводящее к некорректным результатам.

**Дефект (defect, bug, problem, fault):** недостаток в компоненте или системе, способный привести к сбою или отказу. Может быть в документации, настройках, данных.

**Баг:** несоответствие ожидаемого результата фактическому.

**Сбой:** самоустраняющийся отказ или однократный отказ, устраняемый незначительным вмешательством.

**Отказ:** событие, заключающееся в нарушении работоспособного состояния.

---

### Приоритет и Серьёзность
**Серьёзность:** насколько баг влияет на систему (Critical, Major, Minor, Trivial).

**Приоритет:** насколько срочно нужно исправить (High, Medium, Low).

**Пример:** Critical: система не работает. High: критичный функционал сломан. Medium: есть обходной путь. Low: косметический баг.
